Systems¶

System class¶

class Sandbox.System(x: Optional[float] = None, y: Optional[float] = None, theta: Optional[float] = None, perturb_fun: Optional[Callable] = None, init_fun: Optional[Callable] = None)[source]¶

Every object in a Sandox simulation is an instance of a subclass of the abstract class System. In some cases, this is for conceptual reasons rather than practical ones, e.g. in the case of a DisturbanceSource, which certainly can be considered a system but which doesn’t currently inherit anything from System (although this may well change in a future implementation).

__init__(x: float = None, y: float = None, theta: float = None, perturb_fun: Callable = None, init_fun: Callable = None)[source]¶

Construct System. If either x or y are specified in the call to init, then the system will have position and will keep a history of both its x- and y-coordinates over time. If theta is specified in the call to __init__, then the system has orientation, and will keep a history of its orientation over time.

Parameters

  • x (float) – The system’s x-coordinate. Defaults to None.

  • y (float) – The system’s y-coordinate. Defaults to None.

  • theta (float) – The system’s angular orientation. Defaults to None. In Sandbox, orientations are in radians.

  • perturb_fun (function) – A placeholder for a function which can be used to perturb the system’s state. This will typically be used at the beginning of simulation runs.

  • init_fun (function) – A placeholder for a function which can be used to set the initial state of the system in each simulation run.

get_data() → Dict[str, Union[float, List[float]]][source]¶

A function to get the data from a System, in the form of a string-keyed dict. If a System has position, then its current coordinates plus their histories will be included in the data. If a System has orientation, then its current orientation and its orientation history are incuded in the data.

These data, as and when they are included in the returned dict, can be accessed with the following keys:

  • current x-coordinate: data["x"]

  • history of x-coordinates over time: data["xs"]

  • current y-coordinate: data["y"]

  • history of y-coordinates over time: data["ys"]

  • current orientation: data["theta"]

  • history of orientations over time: data["thetas"]

Returns

The System’s data.

Return type

dict

get_data_and_reset() → Dict[str, dict][source]¶

Reset System to its original state and return its data.

init_conditions() → None[source]¶

A placeholder for a method which can be used to set the system’s initial state. An example of this is: an init_conditions method for a mobile agent defines lists of x, y, and theta values. Every time the agent is simulated, in a set of runs, the next set of x, y, and theta values is used, so that the agent’s position and orientaton are varied in a controlled way.

init_theta(theta: float) → None[source]¶

Set the systems initial orientation to the passed in value.

Parameters

theta (float) – The system’s orientation.

init_xy(x: float, y: float) → None[source]¶

Set the systems initial x- and y-coordinates to the passed in values.

Parameters

  • x (float) – The system’s x-coordinate.

  • y (float) – The system’s y-coordinate.

perturb() → None[source]¶

A placeholder for a method which can be used to perturb the state of the system. An example of how this might be used is: every time a mobile agent is simulated, in a set of runs, its pose, [x, y, theta], is shifted by some random amount so that it doesn’t always start from exactly the same place or orientation.

reset() → None[source]¶

Reset System to its original state upon its construction, e.g. so that it can be re-used in another simulation run.

step(dt: float) → None[source]¶

Step the System forwards in time. Subclasses of System will generally override this method, to implement class-specific functionality, but they will also need to call this method if they have either position or orientation, as this is where the history of thos variables over time gets updated.

Parameters

dt (float) – The interval of time to integrate the system over. Currently unused here, but will often be used in subclasses.
